## 一、项目简介

### 1. 项目背景

随着我国医疗卫生体制改革持续深化、"互联网 + 医疗健康"战略全面推进，以及居民健康需求快速升级，传统医疗挂号模式已难以匹配现代化医疗服务体系的效率要求与群众就医体验期待。在人口老龄化加剧、医疗资源分布不均、门诊流量持续攀升的现实背景下，构建基于云计算、大数据、移动互联网的一体化云挂号系统，成为医疗机构数字化转型、优化就医流程、提升公共医疗服务能力的关键举措。

从政策层面看，国家高度重视医疗信息化与便民就医服务建设。国务院办公厅印发《关于促进"互联网 + 医疗健康"发展的意见》，明确提出创新服务模式、提高服务效率、降低服务成本，鼓励医疗机构拓展线上服务渠道。国家卫健委持续推进"改善就医感受、提升患者体验"三年行动，要求二级以上医院全面普及预约诊疗，推动号源统一管理、线上线下一体化服务。医保部门同步完善"互联网+"医疗服务价格与支付政策，为线上挂号、在线结算等服务提供制度保障。一系列政策为云挂号系统的建设与落地提供了顶层设计与合规依据，推动医疗服务从线下为主向线上线下融合转型。

东软云医院挂号系统的建设，旨在解决基层医院信息化薄弱、挂号效率低、患者体验差、数据不通、运维困难等现实难题，助力医院降本增效、规范管理、提升服务质量；同时推动医疗资源合理配置，引导患者有序就医，减少院内聚集，遏制倒号乱象，促进医疗服务公平可及。系统严格遵循国家电子病历、互联互通、数据安全与隐私保护等标准规范，已在全国大量二级医院、县域医共体及基层医疗机构成功落地，成为支撑我国基层医疗数字化转型、落实健康中国战略的重要信息化基础设施。

## 二、系统设计

### 1. 功能设计

#### 1) 用户注册与认证模块

- 手机号 + 密码注册登录
- 注册时录入完整信息（姓名、手机号、密码、邮箱、性别、出生日期、身份证号、籍贯、地址）
- 注册成功后自动将本人添加为默认就诊人
- 就诊人管理（添加、编辑、删除、设为默认就诊人）

#### 2) 医院/科室/医生查询模块

- 左右分栏科室浏览：一级科室列表 + 二级科室/子科室
- 科室图标与颜色标识区分
- 按科室查询医生列表
- 日期横向滑动选择（7天范围）
- 上午/下午时段切换、职称筛选
- 医生信息展示：姓名、职称、擅长领域、号源余量

#### 3) 在线预约挂号模块

- 医生详情页展示排班信息（日期 + 上午/下午时段 + 余号）
- 预约确认页：就诊人选择、费用明细、支付方式选择
- 预约成功后显示确认页面
- 预约记录查看（用户所有预约列表）
- 取消预约功能

#### 4) 个人中心模块

- 用户信息展示（手机号、身份证号脱敏、性别、出生日期、籍贯、地址、邮箱）
- 修改密码（弹窗式）
- 就诊人管理入口
- 我的预约入口
- 退出登录

#### 5) 启动引导模块

- 启动页展示品牌标语与医疗主题插画
- 首页综合服务大厅：搜索框、滚动公告栏、挂号入口大卡、科室九宫格导航、热门专家推荐

### 2. 原型设计

#### 2.1 启动页

居中展示医疗主题插画（🏥❤️📅），标语"专业分诊，轻松挂号"，底部"开始预约"按钮，右上角"跳过"入口。

#### 2.2 登录页

输入手机号、密码，登录成功后跳转首页。底部"注册新用户"入口。

#### 2.3 首页

顶部导航栏（医院名称 + 消息铃铛 + 用户头像入口），搜索框，滚动公告栏，渐变蓝"预约挂号"大卡，在线问诊/体检预约双栏入口，4×2 科室九宫格，热门专家横向滑动卡片。

#### 2.4 科室列表

左侧一级科室纵向列表（高亮选中带指示条），右侧二级科室/科室图标列表。点击科室进入医生列表。

#### 2.5 医生列表

顶部当前科室路径，日期横向滑动选择器（今天+6天），上午/下午/全部切换，有号优先排序。医生卡片：头像、姓名、职称、擅长、余号绿色胶囊、费用、预约按钮。

#### 2.6 医生详情

医生信息大卡片（头像、姓名、职称、从业年限），擅长领域标签，排班时段列表（时间段 + 余号/总号 + 预约/约满按钮）。

#### 2.7 预约确认

步骤条（选择时间→确认信息→完成），就诊信息摘要，就诊人可切换选择，费用明细，支付方式（微信/支付宝/医保），确认支付按钮，支付成功页。

#### 2.8 个人中心

顶部返回栏，居中头像 + 用户名，信息卡片（手机号、身份证脱敏、性别、出生日期、籍贯、地址、邮箱），修改密码弹窗，就诊人管理入口，我的预约入口，退出登录。

#### 2.9 就诊人管理

就诊人列表（姓名、手机号、关系、默认标记），添加/编辑/删除功能，列表返回时自动刷新。

#### 2.10 注册页

全字段录入表单：姓名*、手机号*、密码*、邮箱、性别（单选）、出生日期、身份证号、籍贯、地址。注册成功后自动添加为就诊人。

- **代码质量**：统一实体类驼峰命名规范，添加 MyBatis @TableField 映射，前端 Model 与后端 JSON 字段对齐，消除 snake_case 混用问题。

- **版本管理**：通过 Git 管理代码，上传 GitHub，成功向原仓库提交 Pull Request。

### 2. 遇到的困难与解决方案

**问题1：MyBatis 下划线转驼峰导致字段全为 null**

后端实体类使用 snake_case 字段名（如 `schedule_id`），MyBatis Plus 全局配置 `map-underscore-to-camel-case=true` 会将数据库列名转为驼峰（`scheduleId`）后查找 setter `setScheduleId()`，但 Lombok 为 snake_case 字段生成的是 `setSchedule_id()`，导致映射失败，接口返回全部 null。

**解决**：将所有实体类字段统一改为驼峰命名（如 `scheduleId`），并用 `@TableField("schedule_id")` 显式指定原数据库列名，确保 MyBatis 映射正确。

**问题2：前端页面返回后数据不刷新**

就诊人管理页面从 `PatientEdit` 添加就诊人后 `router.back()` 返回，但 `aboutToAppear()` 只在页面首次加载时触发，不会再次请求数据，导致新增的就诊人不显示。

**解决**：添加 `onPageShow()` 生命周期回调，每次页面重新显示时重新加载数据列表。

**问题3：注册时空邮箱导致后续用户注册失败**

前端注册表单未包含邮箱字段，但提交时传了 `email: ""`。后端只检查了 `getEmail() != null`，未排除空字符串。第一个用户以空邮箱入库后，第二个用户再用空字符串注册就触发"邮箱已被注册"错误。

**解决**：前端注册页增加邮箱输入字段，后端增加 `!email.isEmpty()` 判断，排除空字符串重复检查。

**问题4：router.pushUrl 传复杂对象丢失数据**

从医生列表跳转到医生详情时，将完整的 Doctor 对象作为路由参数传递。在 HarmonyOS 中复杂对象可能因序列化问题丢失字段，导致医生详情页信息为 `undefined`。

**解决**：改为只传基本类型参数（doctorId、doctorName 等），医生详情页根据 ID 调用 API 加载数据。

**问题5：前后端 JSON 字段名不一致导致数据展示异常**

后端 Jackson 序列化 Java 实体时输出 snake_case JSON 键名（如 `time_slot`），而前端 ArkTS Model 使用驼峰命名（如 `timeSlot`），`JSON.parse` 后属性名不匹配导致数据展示为 `undefined`。

**解决**：将后端所有实体类字段统一改为驼峰命名，前后端 JSON 字段名完全对齐。

### 3. 不足与改进方向

**不足**：
